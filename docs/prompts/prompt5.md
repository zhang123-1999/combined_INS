# 任务：审查并修复 INS+UWB ESKF 中 NHC / ZUPT 约束导致速度反向与发散的问题

## 背景
我在一个 C++/CMake 工程中实现了 ESKF（误差状态卡尔曼滤波）用于 INS + UWB 松组合导航。当前正在加入两类约束：
- NHC（Non-Holonomic Constraint，非完整约束）：车体侧向/垂向速度为 0
- ZUPT（Zero Velocity Update，零速修正）：静止时速度为 0

加入 NHC 后，速度曲线出现大量与真值相反的现象；在我把 NHC 的姿态雅可比符号反过来后（StepA），系统出现数百 m/s 级速度发散。

请你对现有工程代码进行“以坐标系一致性为核心”的审查，找出导致 NHC/ZUPT 反向或发散的根因，并给出可编译的修复补丁。

## 工程关键文件（必须重点审查）
- initialization.cpp：初始化状态（p/v/q）与协方差
- ins_mech.cpp：惯导机理传播（坐标系、重力、地球自转、更新顺序）
- eskf_engine.cpp / eskf.h：ESKF 预测/更新、创新定义 y、dx 注入方式、误差状态定义
- measurement_models_uwb.cpp / uwb.h：UWB/NHC/ZUPT 量测模型、雅可比 H、量测噪声 R
- pipeline_fusion.cpp：主循环调度，何时触发 NHC/ZUPT，阈值/门限策略
- data_io.cpp：真值/IMU/UWB 数据读取与字段含义（尤其 quaternion 与 velocity 的坐标系）

## 必须完成的审查目标（逐项给出证据）
### A. 明确全链路坐标系
1. 名义状态 state.p/state.v 是在什么坐标系（ECEF/NED/ENU）？
2. 姿态 state.q 表示的旋转是哪一种：R_nb（n<-b）还是 R_bn（b<-n）？右手系？
3. 真值文件 POS_converted.txt 的 p/v/q 各自在什么坐标系？是否与 INS 传播的坐标系一致？
4. 若不一致，请给出正确的转换关系，并指出工程中目前“直接使用真值 q/v”会产生什么后果（尤其在加入 NHC 后强耦合导致反向/发散）。

### B. 检查 ESKF 更新方程实现是否与量测模型定义一致
我当前实现类似：
- 计算创新 y
- 计算 K
- dx = K * y
- InjectErrorState(dx)

请你逐行确认：
1. y 的定义是否是 z - h(x_nom)（创新/残差）？
2. measurement_models_uwb.cpp 中 NHC/ZUPT 返回的 y 是否与上面定义一致？
3. H 是否是 h 对名义状态的雅可比（∂h/∂x_nom）？
4. 若存在“符号约定不一致”，请指出是哪一处导致正反馈或反向。

### C. NHC 量测模型审查（重点）
当前 NHC 逻辑大致为：
- v_b = R_nb^T * v_n
- z = 0
- y = -[v_b.y, v_b.z]
- H_v = R_nb^T 的 y/z 行
- H_theta = (±)Skew(v_b) 的 y/z 行（曾做过符号试验导致发散）

请你：
1. 推导并核对 H_theta 的符号与误差姿态定义（右扰动/左扰动）及注入方式（q <- q ⊗ δq 或 δq ⊗ q）。
2. 判断 StepA 改符号为何会导致发散（正反馈），并给出正确符号与原因。
3. 判断更根本的问题是否是 v 与 R 使用了不同参考系（例如 ECEF 速度 + NED 姿态），并给出修复方案：
   - 方案1：统一 state.q 为 ECEF<-b
   - 方案2：在 NHC 内部显式引入 ECEF->NED/ENU 的旋转，将 v_ecef 转为 v_local，再施加 NHC

### D. ZUPT 量测模型审查
1. ZUPT 应约束的是速度（通常 local 或 ECEF 速度）为 0。请核对当前工程中 ZUPT 的观测方程和 H 是否正确。
2. 检查静止检测门限（加速度/角速度）是否存在误触发，导致在运动时强制速度为 0 从而拉崩。
